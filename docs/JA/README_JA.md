# Git-Brandmauer

**Languages:** [🇬🇧 English](../../README.md) | [🇷🇺 Русский](../RU/README_RU.md) | [🇯🇵 日本語](README_JA.md)

**Git-Brandmauer** は、**安全な Git リポジトリ管理とネットワーク制御のためのモジュラーフレームワーク** です。

以下の機能を提供します。

- **対話型メニュー** : リポジトリの管理、リポジトリモード (SAFE、NORMAL、OPEN) の設定、ネットワーク操作の実行。
- **状態認識型リポジトリモード** (`~/.git-security/state/` に保存)。これにより、制限モードでは `git push` などの操作が自動的にブロックされます。
- **スマートな Git フックのインストール** (`smart-install-hooks.sh` により、変更されたフックのみを更新し、ユーザーの変更は保持されます。
- **ネットワーク制御スクリプト** (`net/`) : ネットワークの状態確認、接続の一時停止、緊急シャットダウンのトリガー。
- **ログシステム** (`~/.local/share/brandmauer/logs/` 内)。セキュリティイベント、ネットワークアクション、フックの適用を追跡します。
- **モジュラーディレクトリ構造**:
- `LIB_DIR` — 静的スクリプトとライブラリ (`/usr/local/lib/brandmauer`)
- `SECURITY_ROOT` — ローカル状態、フック (`~/.git-security`)
- `LOG_DIR` — ログ (`~/.local/share/brandmauer/logs`)
- `NET_DIR` — ネットワーク操作スクリプト
- **セキュリティ強化** リポジトリモードを考慮し、すべてのアクションをログに記録するフック (pre-push、pre-commit) による強化。

Git-Brandmauer は、単なるスクリプト群ではなく、**完全なシステム** になりました。リポジトリセキュリティ、ネットワーク管理、フック自動化、ログ記録を単一のモジュラーフレームワークに統合しています。

## システムレイアウト (v1.0 安定版)

### 概要

brandmauer は、Ubuntu ベースの環境向けに設計されたシステムレベルのセキュリティツールです。
ファイルシステムレイアウトは、ファイルシステム階層標準 (FSS) および一般的な Ubuntu の規約 (Canonical Ltd. のエコシステム) に準拠した原則に従っています。

バージョン 1.0 では、**安定した固定ディレクトリ構造** が定義されています。

このレイアウトは、今後のマイナーリリースでは変更されません。

## インストールパス

### 1. CLI エントリポイント
```bash
/usr/local/bin/brandmauer
```

- 単一の実行可能エントリポイント
- ディスパッチャとして動作
- ビジネスロジックは含まれません

### 2. コアシステムライブラリ
```bash
/usr/local/lib/brandmauer/
```

内部構造:
```bash
lib/brandmauer/
├── core/ # 基本ユーティリティ、ログ出力、共有関数
├── git/ # git-brandmauer サブシステム
├── net/ # net-security サブシステム
├── hooks/ # git フックテンプレート
├── ui/ # メニュー、出力フォーマット、色
└── config.sh # グローバル設定
```

### 安定性保証 (v1.0)
以下は安定しているとみなされます:
ディレクトリ名 brandmauer
場所`/usr/local/lib/brandmauer`
サブディレクトリ:
* `core`
* `git`
* `net`
* `hooks`
* `ui`
将来的にモジュールが追加される可能性がありますが、既存のパスは変更されません。

### 3. 共有データ (オプション)
```bash
/usr/local/share/brandmauer/
```

予約済み:
* テンプレート
* 静的アセット
* ドキュメント スニペット
このディレクトリは使用されていない場合は存在しない可能性があります。

### 4. ユーザー状態ディレクトリ
```bash
$HOME/.git-security/
```

保存対象:
* リポジトリモード (SAFE / HARD)
* 登録済みリポジトリリスト
* 実行時状態
ユーザーデータは確認なしに削除されることはありません。
この場所は安定しており、公開システム設計の一部です。

## アーキテクチャモデル

内部フロー:
```bash
CLI → Core → Git / Net → Hooks
```

各項目:
* CLI = インターフェース層
* Core = 共有ロジック
* Git / Net = 機能サブシステム
* Hooks = リポジトリ適用層

## セキュリティモデル (v1.0)

### 概要
brandmauer は、Git 操作のためのローカルリポジトリレベルのコマンド制御層を実装します。

その目的は、GitHub でホストされ、Git を介してローカルに管理されているリポジトリにおける、偶発的または安全でないリモート操作のリスクを軽減することです。
このシステムはリポジトリごとに動作し、潜在的に危険なコマンドを許可または制限するセキュリティモードを適用します。

### コアコンセプト
登録された各ローカルリポジトリには、セキュリティモードが割り当てられています。
`OPEN` — リモート操作が許可されます
`SAFE` — 特定のリモート操作がブロックされます
モードはローカルに保存され、自動的に適用されます。

### 制御対象コマンド
SAFEモードでは、以下のGitコマンドが制限されます。
* `git push`
* `git pull`
* `git fetch`
* `git rebase`
* `git merge`
これらのコマンドは、以下の動作を行うため、潜在的に危険であると見なされます。
* リモート履歴の変更
* リモートの変更の同期
* コミット履歴の書き換え
* 外部状態をローカルブランチにマージ

### 適用レイヤー

セキュリティ適用は、以下の方法で実装されます。
* リポジトリレベルの制御ロジック
* Gitフックの統合（該当する場合）
* コマンドインターセプトメカニズム
保護は、リポジトリ全体ではなく、登録済みリポジトリごとに適用されます。
これにより、以下のことが可能になります。
* 重要なリポジトリに対する厳密な保護
* 開発リポジトリに対する柔軟な操作

## モードの動作

### OPENモード
* すべてのGitコマンドが通常どおり動作します。
* 制限は適用されません。

### SAFEモード
* 制御対象コマンドはブロックされます。
* ユーザーには、制限内容を説明する明確なメッセージが表示されます。
* ローカル操作（追加、コミット、ステータスなど）は影響を受けません。

### 保護範囲
brandmauer は、以下の動作を行います。
* ローカルマシンのみ
* 登録済みリポジトリのみ
* Git コマンドレベルのみ
以下の動作は行いません。
* GitHub サーバー設定の変更
* 認証の妨害
* Git 自体の置き換え
* リモート側のポリシーの適用

### 設計原則
1. ローカルファーストのセキュリティ
2. リポジトリごとの制御
3. 最小限の侵入
4. 明示的なモード認識
5. ユーザーによる適用制御

### 脅威モデル（実践的）
このシステムは、以下の影響を軽減するように設計されています。
* 保護されたブランチへの誤った Git プッシュ
* 意図しない履歴の書き換え
* 衝動的なリモート同期
* 機密性の高い開発フェーズにおけるワークフローのミス
以下の攻撃に対する防御は意図されていません。
* 悪意のあるシステムレベルの攻撃
* ユーザーアカウントの侵害
* サーバー側の侵害

## ネットセキュリティサブシステム

net モジュールはシステムの不可欠な部分であり、v1.0 ではオプションではありません。
含まれるツール:
* panic.sh
* burn-zip-archives.sh
* passwd_offline.sh
* net_check.sh
* net-pause.sh
これらのツールはコアセキュリティユーティリティとみなされ、システム設計の一部です。

## アップグレードポリシー

マイナーアップデート (v1.x) ではファイルシステムのレイアウトは変更されません。
メジャーバージョンアップ (v2.0 以降) では構造的な変更が行われる場合があります。
インストールおよびアンインストールスクリプトはこのレイアウトに依存しています。

## 設計理念

brandmauer は、以下の機能を実現するシステムレベルのセキュリティレイヤーです。
* Git リポジトリの制御
* ネットワーク状態の制御
* 緊急時のセキュリティアクション
これはローカルの Ubuntu ベースマシンを対象としており、モバイルやクロスプラットフォームでの展開には適していません。
